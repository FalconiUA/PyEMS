Probes each device: TCP reach → Modbus session → per-register read.

Connection troubleshooter. For every device in this config it checks, in order: (1) can the Pi open a TCP socket to the device IP/port, (2) can it open a Modbus session, and (3) what each profile register actually returns — the raw register words, the decoded value, or the exact Modbus exception code. This is how you tell an IP/port/subnet problem (step 1 fails) from a slave id / address / register-map problem (step 1–2 pass, reads fail).

It probes the devices currently shown on Device profiles / Site YAML (no save needed) — fix host, port or slave id there and re-run. Opening this competes with a running EMS for the same bus: stop the EMS before probing a real device.
